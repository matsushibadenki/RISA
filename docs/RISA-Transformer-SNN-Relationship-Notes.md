# RISA と Transformer / SNN の関係メモ

## 1. 目的

この文書は、RISA と Transformer、および SNN の関係性を整理するための設計メモです。

特に次の問いに答えることを目的とします。

- RISA は Transformer と競合するのか
- RISA は SNN と競合するのか
- 3 方式をどう組み合わせるのが自然か
- 将来的に RISA が SNN の一部役割を取り込める可能性はあるか

---

## 2. 現時点の全体整理

現時点では、次の理解がもっとも自然です。

- **Transformer と RISA は競合しにくい**
- **SNN と RISA は一部競合し、一部補完する**

その理由は、主に解決しようとしている問題が異なるからです。

| アーキテクチャ | 主な目的 | 主な計算対象 |
| --- | --- | --- |
| Transformer | パターン認識・生成 | ベクトル |
| SNN | 時間情報・イベント処理 | 発火 |
| RISA | 世界モデル形成・構造進化 | 関係構造 |

一言で表すと、

- Transformer は「入力から出力を求める機械」
- SNN は「時間変化する信号を処理する機械」
- RISA は「世界そのものを育てる機械」

です。

---

## 3. Transformer との関係

Transformer は、RISA にとって最も自然な **知覚器（Perception Engine）** 候補です。

例えば、

```text
画像
  ->
Transformer
  ->
犬が走っている
```

のような高次元入力の構造化は Transformer が非常に得意です。

一方で、その後に

```text
犬が走った
  ->
疲れる
  ->
休む
  ->
回復する
```

のような世界モデルへ統合し、長期的に構造を更新し続ける役割は RISA が担う方が自然です。

### 基本構成

```text
画像 / 音声 / テキスト
  ->
Transformer
  ->
イベント
  ->
RISA
  ->
世界モデル更新
```

### 含意

- Transformer は知覚と表現抽出に強い
- RISA は意味統合・概念形成・構造更新に強い
- 役割がかなり分かれているため、直接競合しにくい

---

## 4. SNN との関係

SNN と RISA は、Transformer より近い位置にあります。

その理由は、どちらも **イベント** を扱うからです。

ただし、扱うイベントの粒度が違います。

### SNN

```text
Neuron A
  ->
Neuron B
```

発火そのものを処理する。

### RISA

```text
犬
  ->
運動
  ->
疲労
```

意味を持つ関係構造を処理する。

つまり、

- SNN は「情報がどう流れるか」
- RISA は「知識がどう成長するか」

を主に扱っていると考えられます。

---

## 5. 役割分担として見ると自然

3 方式を役割で見ると、次の対応がかなり自然です。

### Transformer

- 見る
- 聞く
- 読む

### SNN

- 感じる
- タイミングを処理する
- 連続イベントを時間統合する

### RISA

- 理解する
- 世界モデルを育てる
- 概念と因果を構造化する

この見方に立つと、競合させるより階層化する方が筋が良いです。

---

## 6. 階層構成の案

現時点で最も自然に見える構成は次です。

```text
高次元入力
        │
        ▼
 Transformer
 （知覚・特徴抽出）
        │
        ▼
 SNN
 （時間統合・イベント化）
        │
        ▼
 RISA
 （概念形成・世界モデル・構造可塑性）
        │
        ▼
 推論・計画・説明
```

この構成では、

- Transformer は「何が見えたか」を表現する
- SNN は「いつ・どの順番で起きたか」を統合する
- RISA は「それが世界の中でどういう意味を持つか」を学ぶ

という分担になります。

---

## 7. 生物学的な比喩

厳密な神経科学対応ではありませんが、役割の比喩としては次がわかりやすいです。

```text
目・耳
  ->
Transformer

視覚野・聴覚野
  ->
SNN

海馬・前頭前野
  ->
RISA
```

もちろん実脳はもっと複雑ですが、

- 知覚
- 時間統合
- 概念・計画

という層分けのイメージとしては有用です。

---

## 8. 競合する部分

SNN と RISA が競合する部分もあります。

それは **継続学習** です。

両者とも、

```text
学び続ける
```

ことを目指せます。

ただし更新対象が違います。

### SNN

```text
シナプス
```

を変える。

### RISA

```text
概念構造
```

を変える。

つまり競合しているようで、実際には **異なるレベルの可塑性** を担っているとも言えます。

---

## 9. RISA が SNN の上位層になる可能性

現時点では、RISA は SNN の代替というより **上位層** になりやすいと考えられます。

つまり、

```text
SNN
  ->
RISA
```

という関係です。

SNN がイベント列を時間的に整形し、RISA がそれを概念構造へ変換する。

この構成なら、

- SNN の時間処理
- RISA の意味構造化

を両立できます。

---

## 10. 逆に、RISA が SNN を部分的に包含する可能性

一方で、より挑戦的な仮説もあります。

もし RISA のエッジ自体が、

- 時間情報
- 遅延
- 頻度
- 因果順序

を明示的に持てるなら、

```text
犬
 --(Δt=0.8s)--> 走る
 --(Δt=15s)--> 疲労
 --(Δt=5min)--> 回復
```

のように、時間も構造の一部として表現できます。

この場合、

- SNN の時間処理
- RISA の構造処理

が一体化し、RISA が SNN の一部役割まで取り込む可能性があります。

### ただし注意

これは現時点ではかなり挑戦的な仮説です。

SNN には、

- イベント駆動
- 低消費電力
- 時間符号化

という強い特徴があります。

RISA がそれらを同等以上に実現できるかは未検証です。

---

## 11. 研究初期の方針

研究初期段階では、Transformer / SNN / RISA を競わせるより、

> 異なる計算原理として組み合わせ、その中で RISA がどこまで独立した役割を担えるか

を検証する方が良いと考えられます。

理由:

- それぞれの長所を活かせる
- RISA 単体の役割を明確にしやすい
- 無理に高次元知覚や時間処理を背負わせずに済む
- RISA 独自の価値を評価しやすい

---

## 12. 実装への含意

この整理から、近い将来の設計に対して次の含意があります。

### 12.1 MVP 系では

- RISA は構造化イベント入力を受ける前提を維持する
- 高次元知覚は外部知覚器へ委ねる
- 時間情報はまず簡易メタデータとして持つ

### 12.2 将来拡張では

- Transformer からイベント変換するインターフェース
- SNN 的な時間統合層との接続
- RISA エッジへの遅延・頻度・順序メタデータ導入

補足:

SNN 的層を RAG のような retrieval と見るより、RISA 内部世界モデルの動的キャッシュ / 再活性化層として見る整理も有力です。この観点は [RISA と RAG、および SNN のキャッシュ類比メモ](RISA-RAG-and-SNN-Cache-Analogy-Notes.md) にまとめます。

を検討する価値があります。

---

## 13. 結論

現時点では、

- Transformer と RISA は競合より補完
- SNN と RISA は一部競合しつつ補完

という理解が最も自然です。

特に有力なのは、

```text
Transformer
  ->
SNN
  ->
RISA
```

という階層構成です。

ただし将来的には、RISA が時間情報を構造の中へ十分取り込めれば、SNN の一部役割まで包含する可能性もあります。

研究初期では、各方式を競争させるより、

> どの役割をどの方式に担わせると最も実用的な知能系になるか

を見極めることが重要です。
